# The verb record format

One verb = one json file at `<store>/<app>/<id>.json`. The directory IS the database — no index to corrupt, and the repair loop's edits show up as normal human-sized diffs. Schema source of truth is [`crates/verbivore-verb`](../crates/verbivore-verb/src/lib.rs) (`format_version: 1`); this doc explains the WHY per field, not a second copy of the types.

A real record, generated by the crawler and accepted by review:

```json
{
  "format_version": 1,
  "id": "submit-login",
  "intent": "submit login",
  "app": "localhost-42002",
  "start_url": "http://localhost:42002/user/login",
  "status": "accepted",
  "steps": [
    {
      "action": { "kind": "type" },
      "target": {
        "container": "login form",
        "intent": "the username field",
        "selector": { "role": "textbox", "name": "Username or Email Address" }
      },
      "text": "verbivore",
      "expect": "dont-care"
    },
    {
      "action": { "kind": "click" },
      "target": {
        "container": "login form",
        "intent": "submit button",
        "selector": { "role": "button", "name": "Sign In" }
      },
      "expect": "change"
    }
  ],
  "assertions": [
    { "kind": "element-gone", "selector": { "role": "button", "name": "Sign In" } }
  ],
  "provenance": {
    "created_at": "2026-07-23T18:41:02Z",
    "source_url": "http://localhost:42002/user/login",
    "grounded_by": "crawler-a11y@v1"
  },
  "variants": [
    {
      "context": { "viewport_w": 1280, "viewport_h": 800, "dpr": 1.0, "zoom": 1.0, "color_scheme": "light" },
      "evidence": [
        { "bbox": { "x": 490.0, "y": 300.0, "w": 300.0, "h": 38.0 }, "score": 1.0 },
        { "bbox": { "x": 490.0, "y": 420.0, "w": 120.0, "h": 40.0 }, "score": 1.0 }
      ]
    }
  ]
}
```

## The design calls, field by field

**Records are data, never code.** The executor interprets a flat step sequence; control flow stays in Rust. Behavior data can't express goes through `{"kind": "custom", "name": "..."}` — a name into the executor's registered-action registry. Unregistered names fail at EXECUTION, not validation, so records travel ahead of their implementations.

**`status` is a lifecycle, and acceptance is execution.** `candidate` → `accepted` → `retired`. The only way to `accepted` is a review run that PASSES — a candidate that can't run never becomes a verb. The executor refuses candidates outside review mode. Repair demotes back to `candidate`: grounding changed, a human re-accepts the diff.

**`steps[].target` carries three addresses for one element, coarsest to finest:**
- `container` — the region intent ("login form"), from the accessibility ancestry. Scopes grounding when names collide across regions.
- `intent` — what a human would call the element. This is what the REPAIR loop re-grounds from when the selector rots: rank the intent over the live page's elements, snap the winner in.
- `selector` — role + accessible name, the durable runtime address. `nth` appears ONLY when the page has role+name twins; an unambiguous selector stays robust to unrelated elements coming and going.

**`expect` is the per-step effect gate, and it is deliberately asymmetric.** `change`: the run fails unless DOM/network signals OR the visual pair model saw something — dual silence is a dead click, the exact sabotage this project exists to catch. `no-change`: signals only — proving stillness VISUALLY on an animated page is where the pair model is weakest (a focus ring sits right at its threshold; measured), so a borderline visual score can't veto a signal-verified no-op. `dont-care`: hovers and typing (a value change is a property write, invisible to mutation observers on plain HTML — don't gate on what the labeler can't see).

**`assertions` are task-level postconditions** — checked against the accessibility tree after the last step ("the login form went away"), a richer target than any per-click pixel diff. The crawler proposes none; constraining the task is review's job.

**`provenance.grounded_by` is a STRING, and that's a rule.** `crawler-a11y@v1`, `rank-a11y@v1 score=0.87`, a model checkpoint id — never a framework type. The schema crate has no browser and no ML dependency, so a record written today parses unchanged when the grounding regime is different in 2028. The dependency graph enforces what the convention promises.

**`variants` make rendering an ENVIRONMENT, not part of the verb's identity.** "Submit login" is the same task at 1280×800 light and 375×800 dark; what differs is the evidence — where the grounder saw each target and how confidently, per rendering context. At runtime the execution context is passed IN, and a context with no variant refuses before the browser opens: running an ungrounded rendering would be guessing, and refusing IS the repair trigger.

## Failure vocabulary

Every way a run can break is a typed, serialized value the tooling consumes (`TargetNotFound`, `AmbiguousTarget`, `EffectSilence`, `NoVariantForContext`, `UnregisteredCustomAction`, `AssertionFailed`...) — "it failed" is never the output, "step 1's selector matched nothing, here's the page it happened on" is. See [`Breakage`](../crates/verbivore-executor/src/lib.rs) and the diagnostic bundle it ships with (run report + per-step before/after screenshots).
